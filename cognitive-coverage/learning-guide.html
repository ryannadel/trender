Trender Cognitive Coverage Guide

An interactive teaching guide for the Trender trend-signal extraction CLI.

Why this guide exists. When AI generates code, a developer may accept it without building a mental model of how it actually works. Over time this creates cognitive debt — the code works, but the understanding is missing or flawed. This guide closes that gap for this specific codebase, one concept at a time, with pointers to the exact code that matters.
Default path: beginner + standard

Why This System Exists

Trender answers: which emerging subtopics inside a broad topic are gaining momentum? It searches recent sources, asks GPT to extract specific themes, then turns those extracted themes into a static interactive trend map.

The important design choice is that Trender treats source-backed evidence as the unit of truth. A trend is not just a word count; it is a cluster of extracted topics tied back to source IDs and bucketed by publication date.

Problem solved

Instead of manually searching arXiv, GitHub, and the web, Trender coordinates discovery, enrichment, LLM analysis, trend scoring, and HTML rendering behind one command: trender scan TOPIC.

The Big Picture

Cast of characters

trender/cli.py conducts the scan. trender/discoverer.py finds candidate sources. trender/crawler.py enriches them. trender/analyzer.py asks GPT for themes. trender/trends.py computes movement. trender/renderer.py makes the report. trender/storage.py persists local history.

Mental Model: Think of Trender as a newsroom pipeline. Reporters gather leads, researchers pull the full article text, an editor identifies the storylines, an analyst measures whether each storyline is accelerating, and a designer publishes a readable briefing.

Data Model

The model layer defines the vocabulary every other module shares. Source is input evidence, ExtractedTopic is GPT's interpretation of that evidence, ScanRecord is one saved run, and TrendMap is the computed report-ready view.

trender/models.py — shared entities
@dataclass
class Source:
    id: str
    title: str
    url: str
    source_type: str
    published_at: str
    summary: str = ""
    content: str = ""
trender/models.py — persisted scan shape
@dataclass
class ScanRecord:
    topic_query: str
    generated_at: str
    window: TimeWindow
    sources: list[Source]
    extracted_topics: list[ExtractedTopic]
Mental Model: The dataclasses are shipping labels on boxes moving through a warehouse. Once every module agrees what a Source or TrendTopic looks like, the pipeline can hand data from station to station without translation.
Key insight: ScanRecord stores both raw sources and GPT-extracted topics. Future reports can recompute movement from historical scans without rediscovering or reanalyzing everything.

Core Algorithm / Decision Flow

Step 1 -> cli.scan() validates API key, builds a TimeWindow, and chooses discovery sources
Step 2 -> discover_sources() gathers arXiv papers, GitHub repos, and web items, then dedupes by URL
Step 3 -> Crawler.enrich() fetches web page text or GitHub READMEs and caches HTML
Step 4 -> Analyzer.analyze() sends source blobs to GPT and requires JSON topic output
Step 5 -> Storage.save_scan() writes the raw scan as JSON
Step 6 -> Storage.load_scans() reloads all prior scans for that topic
Step 7 -> build_trend_map() groups topics, buckets sources, computes velocity/direction, and ranks topics
Step 8 -> render_report() writes a static interactive HTML report
trender/cli.py — scan orchestration
with console.status("Discovering candidate sources..."):
    sources = asyncio.run(discover_sources(topic, window, config, options))
with console.status("Crawling and enriching sources..."):
    sources = asyncio.run(Crawler(config).enrich(sources))
with console.status("Extracting trend topics with GPT-5.4..."):
    extracted_topics = Analyzer(config).analyze(topic, sources)
Key insight: GPT is not optional. The CLI checks OPENAI_API_KEY up front, and Analyzer intentionally raises instead of falling back to heuristics.

Crawling & Enrichment

Discovery gives short summaries. Crawling enriches those summaries with full page text or GitHub README content.

trender/crawler.py — enrichment routing
if source.source_type == "arxiv":
    return source
if source.source_type == "github":
    readme = await self._fetch_github_readme(client, source.url)
    if readme:
        source.content = f"{source.summary}\n\n{readme}".strip()
    return source
Mental Model: Crawling is a librarian photocopying the useful pages before the analyst reads them. The cache avoids making a fresh photocopy every time.
Key insight: HTTP failures during normal page enrichment return the original source instead of failing the scan, but discovery API failures can still raise.

Trend Computation

This is the core algorithm. It merges historical scans, filters sources to the requested window, groups topics by normalized name, counts source evidence by date bucket, computes momentum, classifies direction, and ranks.

trender/trends.py — trend map skeleton
relevant_scans = [scan for scan in scans if scan.window.end >= window.start and scan.window.start <= window.end]
buckets = make_buckets(window)
source_by_id = collect_sources(relevant_scans, window)
topic_groups = collect_topic_groups(relevant_scans, source_by_id)
trender/trends.py — momentum and direction
def compute_momentum(time_series):
    midpoint = max(1, len(time_series) // 2)
    early_points = time_series[:midpoint]
    recent_points = time_series[midpoint:] or time_series[-1:]
    early_rate = sum(point.count for point in early_points) / len(early_points)
    recent_rate = sum(point.count for point in recent_points) / len(recent_points)
trender/trends.py — ranking priority
def trend_rank(topic):
    direction_score = {"emerging": 3, "rising": 2, "stable": 1, "fading": 0}.get(topic.direction, 0)
    confidence = topic.source_count * topic.source_diversity * max(topic.relevance_score, 0.01)
    return (direction_score, topic.velocity, confidence, topic.novelty_score)
Mental Model: The trend engine is a weather station. Individual sources are thermometer readings, buckets are hourly snapshots, velocity is the temperature change, and direction is the forecast label.
Key insight: Topic grouping is simple lowercase/whitespace normalization. Similar but not identical GPT topic names may remain separate.

Report Rendering

The renderer embeds the full TrendMap JSON into a Jinja2-rendered HTML page, then browser JavaScript handles filtering, charting, comparison windows, and cards.

trender/renderer.py — render_report()
def render_report(trend: TrendMap) -> str:
    return REPORT_TEMPLATE.render(
        trend=trend,
        trend_json=json.dumps(trend.to_dict(), ensure_ascii=False),
    )
Key insight: The generated report has no backend. Filtering or charting changes happen in trender/renderer.py, not in a web route.

Storage & Persistence

Storage uses topic slugs and timestamps to organize files under data/scans/ and data/reports/. It also writes a latest.html pointer for quick opening.

trender/storage.py — topic slugs and report writes
def slugify(value: str) -> str:
    slug = re.sub(r"[^a-zA-Z0-9]+", "-", value.strip().lower()).strip("-")
    return slug or "topic"

latest = directory / "latest.html"
latest.write_text(html, encoding="utf-8")
Mental Model: Storage is a filing cabinet, not a database. Each topic gets a drawer; each scan and report gets a timestamped folder label.

Configuration, Safety Boundaries, and Time Windows

trender/config.py reads environment variables and a local .env. trender/timeframes.py enforces valid date input. The CLI refuses to run analysis without OPENAI_API_KEY.

if days is not None and (start or end):
    raise ValueError("Use either --days or --from/--to, not both.")

Key insight: .env is ignored by Git, and this guide does not expose its contents. Secrets are read at runtime and should stay local.
